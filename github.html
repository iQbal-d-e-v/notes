Section 1 ‚Äî Initial Setup (Detailed)

This section walks you through the intermediate-level initial setup for Git, starting from opening Git Bash (or Terminal), configuring your identity, and adding recommended global settings. Each step explains why it matters and shows expected output and troubleshooting tips.

1.1 Open Git Bash / Terminal

How:

  • Windows: Start ‚Üí type Git Bash ‚Üí Enter, or right-click a folder ‚Üí Git Bash Here.
  • macOS / Linux: Open Terminal.
Why: Git Bash provides a POSIX-like shell on Windows with Git & SSH tools available; Terminal on macOS/Linux behaves the same.

Verify Git is installed:

git --version

Expected output (example):

git version 2.42.0
Troubleshooting: "command not found"

If you see command not found or similar, Git isn't installed or the PATH isn't configured. Reinstall from git-scm.com, and during Windows install enable "Add Git to PATH".

1.2 Set Your Global Identity (name & email)

Git requires author information for every commit. Configure this once per machine (global) so your commits show a meaningful author. 

git config --global user.name "Sajid"
git config --global user.email "sajid@example.com"

What this does: stores values in your global Git config (usually ~/.gitconfig) so every repo you create on this machine uses them by default.

Verify the settings:

git config --global --list

Example output:

user.name=Sajid
user.email=sajid@example.com
Local vs Global identity

If you need a different identity per project (e.g., work vs personal), run the same commands inside the repo without --global:

git config user.name "Work Name"
git config user.email "you@company.com"

1.3 Why This Step Matters

Every commit records an author. If you skip this, Git will refuse to commit and suggest the commands above. A typical commit header will show: 

commit a12bc34
Author: Sajid <sajid@example.com>
Date:   Fri Sep 5 10:12:00 2025 +0530

    Initial commit: add README

1.4 Recommended Global Configurations (Intermediate)

These improve cross-platform behavior and developer ergonomics.

a) Default branch name

git config --global init.defaultBranch main

Use main instead of the older default master.

b) Line endings (important on Windows)

# Windows
git config --global core.autocrlf true

# macOS / Linux
git config --global core.autocrlf input

Why: prevents cross-OS issues where CRLF vs LF can show as large diffs or cause build problems.

c) Set commit editor (so commit message prompts open your editor)

# Use VS Code as your commit editor
git config --global core.editor "code --wait"

Why: When Git requires a detailed commit message (e.g., on git commit without -m), it opens your editor and waits for you to finish.

d) Credential helper (so HTTPS auth is smoother)

# Windows (recommended)
git config --global credential.helper manager-core

# macOS
git config --global credential.helper osxkeychain

Why: securely caches credentials so you don't type your PAT/username each push. (GitHub requires a Personal Access Token (PAT) for HTTPS pushes, not an account password.)

e) Nice log alias

git config --global alias.lg "log --oneline --graph --decorate --all"

Convenient alias to visualize commit history with git lg.

Check all config origins

To see where each config value came from (system, global, local):

git config --list --show-origin

1.5 Create a Global .gitignore (Optional but handy)

Ignore OS/editor noise across all repos (e.g., .DS_Store, editor folders). 

# create file and add common ignores
echo ".DS_Store" >> ~/.gitignore_global
echo "Thumbs.db" >> ~/.gitignore_global

# tell Git to use it
git config --global core.excludesfile ~/.gitignore_global

1.6 SSH Keys ‚Äî Short Intro (recommended for GitHub)

Using SSH keys avoids entering tokens every time and is the preferred auth method for GitHub. This page only introduces the idea; see your project‚Äôs SSH setup guide for a full step-by-step (generate key, add to ssh-agent, copy public key to GitHub). 

# generate a modern ed25519 key
ssh-keygen -t ed25519 -C "sajid@example.com"

# start agent & add key
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519
Why SSH?

SSH keys are a pair (private + public). You keep the private key safe on your machine and upload the public key to GitHub. When you push/pull, GitHub verifies you via the key pair ‚Äî no password/PAT needed.

1.7 Final Checklist (Run these to confirm)

# verify git
git --version

# verify identity
git config --global user.name
git config --global user.email

# show the most useful configs
git config --global --list | grep -E 'init.defaultBranch|core.autocrlf|credential.helper|core.editor|alias.lg'

If everything prints expected values, you‚Äôre ready to initialize repositories and start committing safely.

Tip: Keep a separate work vs personal identity if you contribute under two different emails ‚Äî configure local repo settings without --global.

7. Connecting to GitHub

On GitHub ‚Üí create New Repository ‚Üí call it todo-app.

üí° Now your project exists both locally and on GitHub.

8. Collaboration (friend helps)

git switch -c task-list
# edit files

git commit -m "feat: add task list"
git push -u origin task-list

On GitHub, your friend opens a Pull Request ‚Üí you review & merge.

9. Keeping Up-to-date

10. Fixing Mistakes

git reset --soft HEAD~1             # undo last commit, keep changes
git reset --hard HEAD~1             # undo & delete changes
git reflog                          # recover lost commits
